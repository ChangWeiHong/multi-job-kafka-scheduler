Learning doc / simplest useful version

Multi-Job Kafka Scheduler

Build a small NestJS service that stores jobs in Postgres, publishes due work to Kafka, runs fake workers for different job types, retries failures, and records execution history. The first version should teach the moving parts without pretending to be a production platform.

Default decision: keep API, scheduler, Kafka producer, and workers in one NestJS app for the first build. After the flow works locally, workers can be split into separate processes without changing the data model.

What We Are Building

API

Create and inspect jobs

Users create email, billing, report, or export jobs. They can query current status and execution history.

Postgres

Source of truth

Jobs, attempts, retry state, and history live in the database. Kafka is not the database.

Kafka

Delivery pipe

The scheduler sends executable work to topic-per-job-type queues. Workers consume and update Postgres.

Learning constraint: fake the job handlers. Email can log a message, billing can sleep for a second, report can generate a fake result, and export can write a fake URL. The important lesson is orchestration.

Non-Goals For Version 1

  • No real email provider, payment provider, PDF generator, or file storage.
  • No cron parser at first. Use recurrence_interval_seconds so recurrence is easy to see.
  • No Kubernetes, distributed tracing, admin dashboard, or complex permissions.
  • No exactly-once promise. Kafka consumers are at-least-once, so handlers must tolerate duplicates.

Architecture

NestJS API

HTTP endpoints

Validate requests, create jobs, and read job history from Postgres.

->
Postgres + Scheduler

Durable scheduling

Every 5 seconds, claim due jobs and create a queued execution attempt.

->
Kafka Workers

Typed execution

Consume topic-specific messages, run fake handlers, and update attempt results.

Request flow:
POST /jobs
  -> insert row into jobs
  -> scheduler sees next_run_at <= now()
  -> scheduler creates job_executions row
  -> scheduler publishes to Kafka topic
  -> worker consumes message
  -> worker updates job_executions and jobs

Topic mapping:
email   -> jobs.email
billing -> jobs.billing
report  -> jobs.report
export  -> jobs.export

job_executions

id uuid primary key

status enum(queued, running, succeeded, failed)
attempt integer
topic text

error_message text nullable
created_at timestamptz
Simple lock rule: when the scheduler claims a job, set status = queued and locked_until = now() + interval '2 minutes'. A later recovery step can return expired queued or running work to retry.

State Transitions

Flow Transition Meaning
Normal one-off job scheduled -> queued -> running -> completed The scheduler publishes the job, the worker succeeds, and no more runs are needed.
Retryable failure scheduled -> queued -> running -> scheduled The worker fails, attempts remain, and the job receives a later next_run_at.
Permanent failure scheduled -> queued -> running -> failed The worker fails and max_attempts has been reached.
Recurring success scheduled -> queued -> running -> scheduled The worker succeeds and the job is scheduled again using its recurrence interval.
User cancellation scheduled -> cancelled Optional for version 1. Cancel only jobs that have not started running.

Scheduler Algorithm

The scheduler is a small loop. It does not execute jobs. It only decides which jobs are due and publishes messages. 

Every 5 seconds:
  1. Find jobs where:
       status = scheduled
       next_run_at <= now()
       locked_until is null or locked_until < now()

  2. Claim a small batch in a database transaction.
       Preferred Postgres pattern:
       SELECT ... FOR UPDATE SKIP LOCKED

  3. For each claimed job:
       attempt = attempts_made + 1
       create job_executions row with status queued
       update jobs set status queued, attempts_made = attempt
       publish Kafka message with job_id, execution_id, attempt

  4. If Kafka publish fails:
       mark the execution failed
       return the job to scheduled or failed depending on attempts

Kafka Message Contract

Keep the message small. The database owns the full job payload, but including the payload in version 1 makes debugging easier. 

{
  "jobId": "9d9c0b68-2fd7-4b76-8f5a-36942775bb26",
  "executionId": "af20fc60-0bf2-45c9-9a20-b5fe86526c52",
  "type": "email",
  "attempt": 1,
  "payload": {
    "to": "student@example.com",
    "template": "welcome"
  }
}
  • Message key: executionId for attempt-level ordering and duplicate detection.
  • Topic: choose from type using one explicit map.
  • Consumer group: one group per worker type, for example email-workers.

Worker Algorithm

When a Kafka message arrives:
  1. Load job_executions by execution_id.
  2. If execution is already succeeded or failed, acknowledge and stop.
  3. Mark execution running and job running.
  4. Run the fake handler for the job type.
  5. On success:
       mark execution succeeded
       if recurring, set job status scheduled and calculate next_run_at
       otherwise set job status completed
  6. On failure:
       mark execution failed with error_message
       if attempts_made < max_attempts, set job status scheduled with backoff next_run_at
       otherwise set job status failed
Kafka acknowledgement rule: commit the Kafka offset only after Postgres has been updated. If the worker crashes before commit, Kafka may redeliver the message. That is expected.

Retry And Recurrence Rules

Case Rule Simple formula
First failure Schedule another attempt if attempts remain. next_run_at = now() + 10 seconds
Repeated failure Use small exponential backoff for learning. delay = 10 * 2^(attempt - 1)
Max attempts reached Stop retrying and keep the error visible. status = failed
Recurring success Schedule the next run after a successful attempt. next_run_at = now() + recurrence_interval_seconds

Idempotency Rules

  • Treat executionId as the idempotency key for the worker.
  • If an execution is already terminal, do not run the handler again.
  • Make fake handlers deterministic enough to test. For example, fail when payload has {"forceFail": true}.
  • Do not update job status from memory. Reload the job row before deciding retry or completion state.

Acceptance Checklist

  • A delayed job does not publish before next_run_at.
  • An immediate job gets published by the next scheduler tick.
  • Each job type publishes to the expected Kafka topic.
  • A successful worker writes a succeeded execution row and updates the job status.
  • A forced failure creates failed execution rows and retries until max_attempts.
  • A duplicate Kafka message does not run a completed execution again.
  • A recurring job schedules its next run after success.
  • GET /jobs/:id/executions shows every attempt in order.

Interview Talking Points

Topic What to say
Why Postgres and Kafka? Postgres is durable state. Kafka is scalable delivery. This avoids losing jobs when workers are down.
Why at-least-once? Kafka may redeliver messages after crashes, so the worker checks execution state before doing work.
How to scale? Run more consumers in each topic's consumer group. Split job types by topic when workloads differ.
How to prevent duplicate scheduling? Claim due jobs inside a database transaction with row locks or an atomic status update.
What happens when Kafka publish fails? The job remains recoverable in Postgres. The scheduler can retry publish or mark the attempt failed.

One Rule To Remember

Kafka moves work, but Postgres decides truth. If the service restarts, the database should still explain what each job is, what happened to each attempt, and what should happen next.
